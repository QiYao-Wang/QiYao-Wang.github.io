Workspace-Bench：面向大规模文件依赖的AI Agent工作空间学习评测基准

📅 June 05, 2026 · Qiyao Wang #Working Agent #Paper #Benchmark
  • Workspace-Bench：首个系统性评测 AI Agent 工作空间学习（Workspace Learning）能力的 Benchmark，聚焦大规模异构文件间的依赖推理
  • 5 种用户画像、74 种文件类型、20,476 个文件，构建高度真实的工作空间环境
  • 388 个任务、7,399 条 Rubric，覆盖 6 大协同能力维度，每个任务平均标注 5.1 条依赖边
  • 15 种 Agent 配置（4 种 Harness × 7 种 Backbone LLM）全面评测，最强组合仅 ~60%，远低于人类专家 + 工具的 80.7%

Intro

近年来，Foundation Model 和 Agent Harness 的快速发展显著扩展了 AI Agent 的操作边界。然而，当前 Agent 在真实工作场景中的表现仍远未达到可靠水平——即便是最先进的 Agent，在 TheAgentCompany 上也仅能完成 24–30% 的任务，在 OfficeBench 上的成绩也仅为 47%。

核心瓶颈在于：真实工作空间中的任务往往要求 Agent 具备 Workspace Learning（工作空间学习）能力——即识别、推理、利用和更新异构文件之间的显式与隐式依赖关系。例如：

  • 撰写销售提案需要协调非结构化的客户档案、历史沟通记录和结构化的行业知识库
  • 更新项目预算需要交叉引用多个电子表格、演示文稿和 CFO 的邮件确认

然而，现有的 Agent Benchmark 在以下方面存在严重不足：

  1. 预设上下文：将任务所需信息预先打包提供，跳过了真实场景中最关键的自主发现相关信息环节
  2. 文件同质化：仅涉及少量文件格式（< 10 种），忽略真实工作中的 70+ 种异构格式
  3. 依赖关系缺失：将文件视为独立实体，不评估跨文件语义关联、版本追溯、隐式约定等依赖推理
  4. 静态环境：不模拟文件系统的动态演化（版本更新、文件移动、命名歧义等）

⇒ 开发一个能够系统评测 Agent 工作空间学习能力的 Benchmark —— Workspace-Bench

工作空间学习的五个阶段

论文提出了 Workspace Learning（工作空间学习）的概念，并将其演进分为五个阶段：

  • L0：数据不敏感执行（Data Insensitive Execution） — Agent 作为被动顾问，接收任务相关数据，仅提供高层操作指导，不直接操作数据
  • L1：用户指定文件执行（User-Specified File Execution） — Agent 依赖用户明确指定文件路径和操作序列，将文件视为孤立实体，缺乏跨文件依赖意识
  • L2：文件间依赖推理（File-to-File Dependency Reasoning） — Agent 开始主动识别文件间的显式和隐式依赖关系，如区分同一文档的不同版本。此阶段到达 编排奇点（Orchestration Singularity）：Harness 对任务执行的贡献开始超过 Foundation Model
  • L3：任务到文件的依赖发现（Task-to-File Dependency Discovery） — Agent 成为主动探索者，根据高层任务意图自主搜索整个工作空间。此阶段到达 能力奇点（Capability Singularity）：Agent 能够端到端独立完成任务。评测显示当前 Agent 在此阶段性能从 Easy 的 57.6% 骤降至 Hard 的 40.5%
  • L4：工作空间原生自进化（Workspace-Native Self-Evolution） — Agent 成为与用户工作空间共同演化的"活体伙伴"，将每次任务执行和环境变化内化为隐式反馈，持续优化自身能力

从 L2 开始，Harness 的贡献持续大于 Foundation Model。在 L3/L4 阶段，所需的工作空间学习能力与当前 Agent 孤立文件处理范式之间的失配不断扩大，形成 数据关联鸿沟（Data Association Gap）——这是当前 AI Agent 尚未能跨越的根本性瓶颈。

Benchmark 构建

Workspace-Bench 的设计遵循四大原则：(1) 高保真关系型工作空间：模拟真实的混乱数字工作空间，包含嵌套目录、语义噪声文件、隐式跨文件依赖；(2) 依赖驱动的推理：显式标注并评估文件间的依赖交互，而非将每个文件视为孤立证据源；(3) 真实任务标注：任务来源于真实办公场景，由领域专家人工标注，LLM 仅作为辅助验证工具；(4) 过程感知的细粒度评估：不仅评估最终输出，还评估中间决策（如是否正确选择了文件版本）。

工作空间构建

构建了五种代表性职业角色的工作空间：运营经理、物流经理、产品经理、后端开发者和研究员，覆盖互联网公司中不同的工作流和文件组织结构。

采用自顶向下的两阶段混合构建流水线：

  • 结构生成（Structure Generation）：首先为每种角色定义详细的画像（职责、工作流、文件使用模式、领域术语），然后基于画像生成反映角色特征的文件目录层次，并引入受控的结构噪声（冗余文件夹、歧义命名、归档目录等）
  • 内容填充（Content Population）：采用混合策略——首先用语义驱动的 Agentic Crawler 遍历目录树并抓取相关公开资源（arXiv 论文、GitHub 仓库、技术文档、报告等），然后用 LLM 基于已有文件合成相关工件（如讨论某篇论文的邮件、引用设计文档的会议记录、基于电子表格派生的分析报告等）

最终由领域专家审核，确保目录结构与角色的匹配性、文件内容与其位置的一致性，以及注入的文件关系能够支撑有意义的任务。构建的工作空间包含三大挑战：任务相关文件检索、版本追溯（如区分 report_v1、report_reviewed、report_final）、跨模态异构源推理。

任务构建

基于构建的工作空间，人工构建了 388 个任务。任务流程包括：

  • 真实工作流收集：通过内部问卷收集真实工作场景中的任务描述、预期输入和期望输出，经领域专家筛选后标准化为 154 个代表性任务场景
  • 多维度任务标注：25 名标注者根据五种角色创建具体任务，为每个任务编写自然语言指令、识别所需输入、生成参考答案、设计评估 Rubric。同时构建文件依赖图谱（File Dependency Graph），标注者显式标注完成任务所需的最少关键文件路径集合
  • 客观可评估性保证：利用辅助 Agent Pipeline 将模糊标准转化为数据锚定的断言（如将"计算是否正确？"转化为"最终值是否等于 [具体值]？"），并由人类专家交叉验证所有任务、依赖图谱、参考答案和 Rubric

每个任务平均需要解析 5.1 条依赖边（跨 4.7 个不同文件），通过平均 19.1 条 Rubric 进行评估（范围 6–30）。Rubric 分为三类：结果导向（54.8%，验证最终输出的正确性和完整性）、基础（25.0%，检查文件命名、格式、存储位置等基本合规性）、过程导向（20.2%，评估推理和执行过程的合理性）。

评估框架

设计了专门的评估系统，解决三个核心挑战：(1) 任务隔离与快速恢复：通过独立 Sandbox 和并行 BFS 工作空间回滚算法，确保跨任务无污染；(2) 结果精确抽取：采用三路并行检索策略——指令约束路径提取、统一副本集中检索、基于元数据的全局模糊匹配；(3) 并行评估加速：工作空间级和任务级双重并行机制，显著提升评估吞吐量。

Benchmark 统计

Workspace-Bench 的核心统计数据：

  • 5 个工作空间（5 种职业角色），总计 20,476 个文件，3,299 个目录，最大目录深度 8 层
  • 覆盖 74 种文件类型：文档（.docx, .pdf, .md）、电子表格（.xlsx, .csv）、演示文稿（.pptx）、代码（.java, .py, .js）、配置（.yaml, .json）、邮件（.eml）、统计数据集（.dat, .sav）、图像（.png, .jpg）等
  • 388 个任务，每个任务平均 19.1 条 Rubric，总计 7,399 条
  • 任务难度分布：Easy 14%、Medium 53%、Hard 33%
  • 6 大能力维度覆盖：工作空间探索（67.5%）、任务支撑文件利用（61.3%）、结果提供文件利用（54.4%）、内容关系理解（43.8%）、语义异构文件理解（36.1%）、版本追溯（35.1%）
  • 额外提供 Workspace-Bench-Lite：100 个任务的精简子集，保持分布一致性，评估成本降低约 70%

实验

实验设置

评估了 4 种 Agent Harness（OpenClaw、LangChain DeepAgent、Hermes）搭配 7 种 Foundation Model（GLM-5.1、GPT-5.4、Gemini-3.1-Pro、Kimi-2.5、MiniMax-M2.7、Grok-4.3、Qwen-3.6-Plus），共 15 种配置。所有评估在 Workspace-Bench-Lite 上进行。

主要结果

整体结果：所有配置的 Rubric 通过率分布在 27%–60% 之间，平均仅为 45.1%，严重低于人类专家水平（80.7%）。表现最佳的组合是 DeepAgent + GLM-5.1，紧随其后的是 Hermes + GLM-5.1 和 OpenClaw + GLM-5.1。

关键发现：拥有更强规划与推理能力的 Foundation Model 通常带来更高的 Rubric 通过率；在相同 Backbone LLM 下，Harness 框架的选择也对最终任务执行性能产生显著影响。

深入分析

发现1：Agent 在执行更高级工作空间任务时性能持续显著下降

从 Easy (51.4%) 到 Medium (46.0%) 再到 Hard (35.7%)，通过率呈阶梯式下降。Easy 任务涉及简单异构输入和较少推理步骤，Hard 任务则要求高级能力——文件关系发现、长程规划、状态跟踪和错误恢复。性能的下降是 基础 LLM 推理局限性与 Harness 编排约束双重作用的结果。GLM-5.1 搭配三种 Harness 均表现出强鲁棒性，而 Hermes + Gemini-3.1-Pro 在 Hard 任务上跌至 30% 以下。

发现2：6 大工作空间任务维度中，3 个维度的能力构成当前 Agent 的主要瓶颈

大多数配置在工作空间探索和结果提供文件利用上表现相对较好（前者得益于 Agent 的工具调用能力，后者依赖 LLM 的推理能力）。异构文件理解（Heterogeneous File Understanding）和版本追溯（Lineage Tracing）在所有 Agent 中持续垫底——跨格式内容解析和深度跨文件依赖推理是所有系统共同面临的通用瓶颈。

更强的基础模型（如 GLM-5.1）在不同协同维度上表现出显著方差：DeepAgent + GLM-5.1 在 Workspace Exploration 上接近 50%，但在 Heterogeneous File Understanding 上降至约 30%。表明 Harness 的编排策略能够重塑同一 Backbone LLM 的能力分布。

发现3：不同 Harness 和基础 LLM 在不同用户画像上表现各异

多数 Agent 配置在后端开发和研究员画像上表现更好（偏重代码执行和结构化数据处理），而在产品经理和运营经理等偏业务型画像上性能下降显著——后者需要战略规划、资源分配和模糊语义理解。Hermes + GLM-5.1 在 Researcher 维度上接近 65%，得益于 Hermes 对代码开发和研究任务的原生优化。

发现4：高交互轮次和高计算成本并不保证优质任务表现，优秀 Agent 系统展现出卓越的"推理效率"

虽然更多交互轮次通常意味着更全面的推理链和更多试错机会，但结果表明显著的反例：Hermes + GLM-5.1 以极低交互轮次（< 30）和最小 Token 消耗实现了超过 55% 的平均准确率，展现出卓越的"推理效率"。而 DeepAgent + GLM-5.1 取得近 60% 的顶级准确率，却以极高 Token 消耗和大量交互轮次为代价。

配置聚集在右下区域的组合（如 DA+Gemini、HM+Gemini）产生 40–60 轮交互、消耗海量 Token，准确率却停滞在 30%–45%。当底层 LLM 缺乏足够复杂的推理和自我反思能力时，Agent 极易陷入无意义的重复循环。

发现5：人类专家与 Agent 协作仍然显著优于全自主 Agent

20 名领域专家在仅提供任务指令和对应工作空间文件、可使用 Agent 作为辅助工具的条件下，Rubric 通过率达到 80.7%，远超所有全自主 Agent 配置。值得注意的是，人类基线的通过率不随任务复杂度增加而显著下降，这归因于专家天生具备辨别异构文件间潜在关系并灵活利用这些联系解决问题的能力。

错误分析

将 Rubric 失败原因分为五类：(1) 约束错误（文件命名/路径违规）、(2) 内容缺失（核心信息遗漏）、(3) 推理错误（统计/聚合/排序/关联/数学计算不准确）、(4) 过程错误（执行轨迹缺陷）、(5) 格式错误（输出结构不符）。

内容缺失和推理错误构成绝大多数失败原因，表明当前 Agent 在复杂文件系统中的主要瓶颈在于深度嵌入信息的全面召回和跨文件数据聚合与理解能力。格式错误和过程错误占比极少，说明现有模型在执行基础工作流和遵循格式化输出指令方面已相对成熟。
